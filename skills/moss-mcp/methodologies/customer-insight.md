# 客户洞察方法论

## 核心方法

客户洞察不是“企业信息大全”。先识别用户到底要解决什么业务问题，再把问题路由到一个 `primary_scene`，只查询该场景需要的维度，把事实转成商业判断和下一步动作。

稳定顺序：

1. 锁定主体：企业全称和 18 位统一社会信用代码优先；品牌、简称、集团、分公司要声明本次口径。
2. 判断 `primary_scene`：不要把所有企业问题都套成企业档案。
3. 选择最小必要数据：按实时 `tools/list` 和 schema 选择当前 Client 授权能力，不假设固定工具存在。
4. 并行查询同一场景内的独立维度；串行处理依赖上一步结论的动作。
5. 将事实提纯为：对客户意味着什么、对我方意味着什么、下一步怎么做。
6. 明确局限：未授权、空结果、数据源失败、主体不确定、窗口未查完都必须写出。

若用户有明确目的、困惑或判断诉求，例如“是否值得跟进”“从哪里切入”“为什么变化”，不要铺企业百科；围绕真实目标组织答案。

## `primary_scene` 路由

| 场景 | 何时进入 | 方法论重点 | 不要误入 |
| --- | --- | --- | --- |
| `company_overview` | 单个企业、企业档案、公司怎么样、销售跟进材料、简短介绍 | 锁定主体后做综合画像；结论先行；补业务切入建议 | 投资亏损归因、股价估值、复杂行业研究、单独风险舆情 |
| `company_scale` | 营收、员工 / 参保人数、注册资本、资产规模、规模变化、多企业规模对比 | 只围绕体量指标；多企业同口径并行对比；非上市财务缺失要说明 | 财务健康度、盈利能力、估值、经营质量深度分析 |
| `company_multidim` | 用户点名专利、股东、诉讼、资质、联系方式、新闻、工商变更等一个或多个维度 | 点名维度并行查询；大数据量只展示摘要和下载 / 后续深查建议 | 纯营收人数规模问题；只要综合画像 |
| `company_similar` | 找相似、对标、同赛道，且有参照企业 | 先查参照企业画像，再明确对标维度，最后转为名单筛选 | 没有参照企业的条件筛选；CRM 相似客户 |
| `company_filter` | 按行业、地区、规模、资质、融资、标签筛企业名单 / 目标客户 | 把自然语言条件转为结构化筛选；分页合并去重；输出名单摘要和收窄建议 | 从自身产品出发问“还能卖给谁”；问合作状态 |
| `chain_prospect` | 从产品 / 服务 / 行业节点出发问下游、潜客、新客群、第二增长曲线 | 先识别产业链节点，再给潜客方向；用户选方向后转名单筛选 | 已经给出明确行业、地区、标签条件的名单筛选 |
| `batch_file_query` | 上传图片 / Excel / 文件，或一次给三家以上企业，或长文本名单 | 批量识别、批量查询、汇总对比；无法识别项和异常项单列 | 单家企业查询 |
| `crm_query` | 问我方合作、合同、责任销售、产品线、客户名单 | CRM 是企业内部能力；未配置时只能说未配置，不能说未合作 | 不涉及合作关系的纯企业信息查询 |

路由优先级：

1. 文件上传或三家以上企业名单 → `batch_file_query`
2. 合作 / CRM / 合同 / 我们和某企业 → `crm_query`
3. 相似 / 对标 / 同赛道且有参照企业 → `company_similar`
4. 按条件要企业名单 / 目标客户 / 推荐企业 → `company_filter`
5. 从产品或服务问下游市场 / 还能卖给谁 → `chain_prospect`
6. 点名一个或多个具体维度 → `company_multidim`
7. 核心问题是规模体量 → `company_scale`
8. 单企业综合画像 / 销售跟进材料 → `company_overview`

## 场景执行要点

### `company_overview`

- 必做：主体识别、基本盘、经营状态、规模代理、行业定位、近期动态。
- 可选：招投标、招聘、知识产权、风险信号，只在用户目的需要时补。
- 输出：一句判断 + 画像摘要 + 商机信号 + 切入建议 + 局限。
- 禁止：把全部可查维度堆成清单；没有证据就写“行业龙头”“发展迅猛”。

### `company_scale`

- 指标优先级跟随用户问题：问营收就先答营收，问人数就先答人数。
- 多企业对比必须统一口径：同一时间范围、同一指标定义、同一数据来源层级。
- 非上市公司没有完整三表时，可用注册资本、参保人数、年报、招投标、招聘等做代理，但必须标注“代理指标”。
- 规模变化要区分真实增长、口径变化、注册资本认缴变化和数据缺失。

### `company_multidim`

- 只查用户点名维度；用户点了多个维度就并行查。
- 联系方式 / 关键人触达要说明来源和可用性，不承诺一定能触达。
- 新闻动态 / 发展状况不等于舆情统计；只有用户明确说舆情、口碑、负面、声量才进入舆情方法论。
- 专利、招投标、诉讼等大数据量维度：先给统计摘要、代表样本和下载 / 后续深查建议，不把大量明细塞进正文。

### `company_similar`

分两步，不要跳过询问维度：

1. 先查参照企业，提取行业、区域、规模、资质、融资上市、成立年限等对标基础。
2. 让用户或上游任务选择对标维度；没有选择时给 3-6 个推荐维度。
3. 将选择转成筛选条件，复用名单筛选逻辑。
4. 输出时先说明“相似”的定义，不要把“同名 / 同行业 / 同区域”直接当相似。

### `company_filter`

自然语言条件转译规则：

- 地域：区县 > 城市 > 省份；精细地域不要重复传上级地域。
- 行业：优先匹配工具支持的行业 / 产业链节点；失败时用近义词、上位词、下位词重试。
- 多个筛选条件通常按 AND；多个同类行业节点通常先 OR，再与地域、规模等条件 AND。
- 规模：注册资本、参保人数、营收、融资轮次不是同一件事；工具缺哪个就说明缺哪个。
- 结果很多时先取总量和代表样本，再提示收窄条件；不要默认深查全部企业。
- 分页结果需要合并、去重、排序；统一社会信用代码优先作为去重键。

### `chain_prospect`

- 先问清“我方产品 / 服务解决什么问题”，再识别产业链节点。
- 输出方向而不是直接承诺名单：潜客行业、购买场景、采购部门、触发事件、进入条件。
- 用户选择方向后，再进入 `company_filter` 拉名单。
- 若产品过于泛化，先给 3-5 个可能客群并询问优先级。

### `batch_file_query`

- 多文件逐个处理，不静默合并。
- 文件或图片识别出的企业名要保留原文、识别结果、置信度和无法识别项。
- 三家以内可直接文本汇总；三家以上优先给统计摘要、异常项、代表样本和文件交付。
- 批量结果中 CRM 未配置、企业未匹配、同名歧义要单列，不能写成无数据或无合作。

### `crm_query`

- 子意图分四类：合作事实确认、合作概况、深度合作分析、客户名单筛选。
- 未配置 CRM 能力时，固定表达为“当前 Client 未提供 CRM 查询能力，无法判断合作记录”，不能写“未合作”。
- 深度合作分析才需要补外部企业画像；简单合作事实不要强行扩展企业背景。
- CRM 数据和公开企业数据要分层呈现，不能混成同一来源。

## 查询范围控制

- 指定维度优先，不自动补全所有企业维度。
- 用户问风险：只补处罚、异常、司法、风险新闻等必要事实，不自动扩展技术、招投标、专利。
- 用户问舆情：优先查舆情 / 新闻，不自动补财务、股东、专利。
- 用户问行业趋势、数字化现状、业务痛点时，不要误用舆情统计；用公开资料或企业动态时要标注来源口径。
- 工具未授权时，换用已授权的近似能力前先说明降级口径。
- 空结果只代表当前工具、条件和窗口无返回，不代表事实不存在。

## 高频误判清单

| 误判 | 正确做法 |
| --- | --- |
| 单企业查询都套 `company_overview` | 先看用户问的是规模、维度、名单、对标、CRM 还是产业链 |
| 新闻动态当成舆情统计 | 新闻只能说明公开动态；声量、情感、危机等级要走舆情方法 |
| CRM 未配置写成未合作 | 只能写当前 Client 未提供 CRM 查询能力，无法判断 |
| 名单筛选结果少就说市场小 | 先检查行业词、地域、规模条件是否过窄，说明筛选口径 |
| 联系方式存在就等于可触达 | 要说明来源、关键人标识和可用性，不承诺一定触达 |
| 参保人数、注册资本、营收混用 | 三者口径不同，只能作为不同规模信号 |
| 大量明细直接塞正文 | 先摘要、分类、代表样本，再给后续深查或文件交付 |

## 风险和舆情的吸收方式

客户洞察可以吸收风险或舆情字段，但不要把整篇客户洞察改写成风险报告或舆情报告。

风险字段最小集合：

```text
risk_level: 高 / 中 / 低 / 未知
trigger_reasons: 触发原因
risk_path: 对合作、账期、准入、投资的影响路径
confidence: 高 / 中 / 低
recommended_action: 复核动作和业务动作
```

舆情字段最小集合：

```text
opinion_level: L1 / L2 / L3 / 未知
sentiment: 正面 / 中性 / 负面 / 混合 / 未知
key_topics: 高频话题
spread_sources: 主要平台或信源
confidence: 高 / 中 / 低
recommended_action: 动作建议
```

吸收规则：

- 风险字段进入合作优先级、账期、准入或下一步动作。
- 舆情字段进入近期动态、市场信号或触达时机。
- 字段未知就保留未知和待补，不硬补结论。

## 洞察输出

默认结构：

1. 结论摘要：一句话回答用户问题。
2. 主体与口径：企业全称、统一社会信用代码、是否含集团 / 分支 / 品牌映射。
3. 关键事实：只列与目标有关的事实，带来源和时间。
4. 商业判断：解释事实对客户、我方、合作动作的影响。
5. 下一步动作：跟进建议、补查建议、收窄条件或名单交付。
6. 局限：未授权、空结果、数据缺口、主体歧义。

常见解释口径：

- 招投标活跃看预算、采购节奏和进入机会。
- 招聘扩张看研发、销售、交付投入。
- 知识产权和资质看壁垒、准入门槛和产品匹配度。
- 产业链位置看上下游依赖和新客群。
- 近期动态看触达时机，不直接等同于舆情风险。
- 风险信号影响优先级、账期、准入和尽调动作。
