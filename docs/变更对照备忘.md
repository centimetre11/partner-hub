## AI 建档（伙伴 / 客户）

### 伙伴 AI 建档不应识别商机（2026-06-22）

**现象 / 常问**  
「AI 建档伙伴的时候，不要识别里面的商机」— 粘贴会议纪要建档时，草案里出现商机区块，或保存时往客户维度写商机。

**根因**  
`new_partner` scope 在 `proposal-scope.ts` 里允许 `opportunities`；`ai-locale` 提示「有 deal 就填 opportunities」；`applyIntake` 会对 proposal 里的商机落库。

**改动点**  
- `src/lib/proposal-scope.ts` — `new_partner` 过滤 `opportunities`，UI 草稿不展示商机区  
- `src/lib/ai-locale.ts` — 提示明确禁止从建档文本抽商机  
- `src/lib/ai-intake.ts` — `scope === "new_partner"` 时跳过商机写入  

**方案摘要**  
伙伴建档只收：公司信息、联系人、待办/培训/方案/商务记录等；商机走独立入口「✦ AI 添加商机」（客户维度）。

**回归注意**  
- **不要**给 `new_partner` 恢复 `opportunities: true`  
- **不要**影响 `opportunity` scope、`list_opportunities` 查询、客户详情页加商机  
- 客户建档 `new_customer` 本来也不抽商机，保持一致  

**最后验证**  
- [x] 2026-06-22 · 生产 · 已部署 commit `05e66ee`

### 仅 KMS 链接建档很慢（~3 分钟）（2026-06-22）

**现象 / 常问**  
「为什么 AI 建档耗时这么久？」— 只发一条 KMS 链接（如 AUAV），也要 3 分钟左右；Token 日志显示 `web_search` ~53s + `linkedin_search` ~33s 串行，且各跟一次增量 LLM。

**根因**  
1. `RESEARCH_GUIDE` 要求 KMS 后仍做 web + LinkedIn  
2. 模型分两轮调用搜索，**串行**执行  
3. 每次搜索后 `extractPatchFromTool` 再调一次 LLM（增量提取）  
4. 主流程末尾还有一次 JSON 提取  

**改动点**  
- `src/lib/intake-public-research.ts` — 识别公司名后 **预并行** web + LinkedIn，结果注入对话；重复调用跳过  
- `src/lib/ai-tool-loop.ts` — 同一轮多个 tool call 改为 `Promise.all` 并行  
- `src/lib/ai-locale.ts` — 提示：已预取则勿重复搜索；需自搜时同轮并发  

**方案摘要**  
公开检索并行，避免「搜网页 → 等 LLM → 再搜 LinkedIn」。KMS 场景下仍保留最终一次 LLM 汇总（未做增量提取改启发式，见下条待办）。

**回归注意**  
- 预取仅用于 `new_partner` / `profile` / `new_customer` / `customer_profile` 且有可识别公司名  
- **fast scope**（`todo`、`powermap`、`business_record` 等）**不能**走调研预取  
- 改 `runToolLoop` 并行时，注意 tool 结果顺序仍按 `tool_call_id` 写回 chat  
- 若改 `MAX_RESEARCH_STEPS`，对照 Token 日志看是否又多轮无效搜索  

**最后验证**  
- [ ] 部署后 · 同一条 KMS 链接 · 目标 <90s（视搜索 API 而定）

### 增量提取 vs 启发式（待做 · 2026-06-22 讨论）

**现象 / 常问**  
「增量提取改为启发式、KMS 场景不走额外 LLM」是什么意思？

**根因**  
每次 `web_search` / `linkedin_search` 完成后，`proposal-patch-extract.ts` 会再调一次 `chatCompletion`（`Incremental extract·*`），KMS 已预读时多为重复劳动（AUAV 案例约 +27s）。

**方案摘要（未全面实施）**  
KMS 预读成功且字段够用时：右侧草稿用 `heuristicPatch` 规则更新，跳过增量 LLM；最终仍保留一次 JSON 提取。

**回归注意**  
若实施：只减 **中间** 增量 LLM，不能省最终 `extractIntakeJson`；复杂长文本建档可能仍需要 LLM 增量。

**最后验证**  
- [ ] 未实现

## 企微 Bot · 协作录入

### 待办标题带命令词、负责人进标题（2026-06-22）

**现象 / 常问**  
`@bot 增加一个待办，update power map，负责人是 areeb` → 草案摘要和标题都是整句，负责人未单独识别。

**根因**  
1. `stripIntakeCommandPrefix` 只匹配「加/创建/新增待办」，**不匹配「增加一个待办」**  
2. `TodoProposal` 无 `assigneeName`，保存时 `assigneeId` 固定为操作人  
3. LLM 偶发把整句写入 `title` / `summary`  

**改动点**  
- `src/lib/todo-intake-parse.ts` — 统一剥离命令前缀、解析 `负责人是 xxx`  
- `src/lib/fast-intake-heuristic.ts` / `src/lib/ai-intake.ts` — 归一化 todo  
- `src/lib/proposals.ts` — `assigneeName` 字段  
- `src/lib/proposal-wecom-format.ts` — 预览展示负责人  
- `src/lib/ai-locale.ts` — todo 提示：标题只写事项本身  

**方案摘要**  
标题 → `update power map`；负责人 → `areeb`（模糊匹配 Hub 用户）。

**回归注意**  
- 覆盖变体：`加待办` / `增加待办` / `create todo` / `add todo`  
- `负责人：` / `责任人：` / `assignee:` 等应在 **assigneeName**，不在 title  
- **不要**在 `business_record` 的 strip 规则里误伤待办  
- 无负责人时仍默认当前操作人  

**最后验证**  
- [ ] 部署后 · 企微群 · 同上例句

### 待办草案显示可确认、@确认 却报不完整（2026-06-22）

**现象 / 常问**  
`记个待办给 Zayne，smc 做出历史最佳标杆` → 预览 ✅ 信息已足够；`@bot 确认` →「草案信息还不够完整」。

**根因**  
1. 企微存 `proposeSessions` 时，**商务记录**会 `refinal` 重算 `ready`，**待办没有** → 预览 `ready` 与 session 可能不一致  
2. 确认前未再次 `finalizeTodoTurn`（绑定群 + 误把负责人当 `partnerName` 时会触发 blocking 澄清）  
3. `给 Zayne，…` 未解析为 `assigneeName`，整句进 title  

**改动点**  
- `src/lib/wecom-bot.ts` — 待办存 session / @确认 前 `refinalProposeIntakeTurn`；propose 时清 `intentConfirmSessions`  
- `src/lib/fast-intake-heuristic.ts` — 导出 `refinalProposeIntakeTurn`  
- `src/lib/intake-partner-binding.ts` — 绑定群写入正确 `partnerName`；Hub 人名不当 partner  
- `src/lib/todo-intake-parse.ts` — `记个待办给` / `给 X，` → assignee  
- `src/lib/hub-assignee-names.ts` — 团队成员名单注入 prompt  
- `src/lib/proposal-wecom-format.ts` — 待办确认 blocked 时展示清单  

**回归注意**  
- 已绑定伙伴群：待办 + 负责人 + @确认 应一次保存  
- 开放式录入：未匹配公司名仍应 blocking，清单可见  
- `给 jackie/areeb` 等应进 assigneeName，不进 partnerName  

**最后验证**  
- [ ] 部署后 · MENA 群 · `记个待办给 Zayne，…` → @确认 成功

### 未绑定群：@机器人 / global 误当伙伴（2026-06-27）

**现象 / 常问**  
开放群 `@MENA Beard Gang 给宋健加待办…` → 归属伙伴显示机器人名；或 `约一下 global 线上` → 误链 **GlobCom**。

**根因**  
1. `@` 剥离正则吞中文，把「给宋健…」当机器人名  
2. LLM / `contains` 模糊匹配把 **global** 当成 **GlobCom**  
3. 开放群未校验 `partnerName` 可信度  

**改动点**  
- `src/lib/wecom-bot-intake.ts` — client-safe `@` 剥离（与 `wecom-user-resolve` 一致）  
- `src/lib/intake-partner-binding.ts` — `sanitizeOpenIntakePartnerName` / `isOpenTodoPartnerNameCredible` / `userTextMentionsPartnerName`  
- `src/lib/wecom-bot.ts` — scope 切换清旧 fields；`sanitizeProposalForScope`  

**方案摘要**  
开放群：只有用户**明示**公司名才关联；`global` 不等于 GlobCom；机器人名 never partnerName。

**回归注意**  
- **不要**在开放群逻辑里去掉 bot 名校验  
- **不要**恢复 `suggestPartnersFromIntakeText` 的前 40 字模糊搜伙伴  
- **绑定群**与开放群是两套规则（见下条）  

**最后验证**  
- [x] 2026-06-27 · 脚本 `test-todo-intake.ts` + `test-intake-use-cases.ts`

### 绑定 GlobCom 群：待办默认挂绑定伙伴（2026-06-27）

**现象 / 常问**  
GlobCom 伙伴群已绑定 chatId，建待办应默认归属 **GlobCom**；正文里的 `global` 只是会议描述。

**根因**  
绑定群仍走开放群推断 / 旧 scope 草案残留（建档 fields 混入待办预览）。

**改动点**  
- `src/lib/intake-partner-binding.ts` — `applyBoundContextToProposal`（群绑定优先）  
- `src/lib/wecom-bot.ts` — propose / @确认 前后强制绑定上下文  
- `src/lib/ai-locale.ts` — 绑定群 todo 提示：勿从 global/online 推断公司  

**方案摘要**  
`chat.partnerId` 存在 → `partnerName` 强制为绑定伙伴；待办 scope 清掉误入的 fields。

**回归注意**  
- 改开放群伙伴逻辑时 **不要** 影响 `boundPartnerId` 分支（`enrichTodoPartnerBinding` 首段 + `applyBoundContextToProposal`）  
- GlobCom 绑定群 + `global` 正文 → 仍必须是 **GlobCom**，不是「无归属」  
- 未绑定群 → 仍必须 **不能** 因 global 链 GlobCom  

**最后验证**  
- [ ] 部署后 · GlobCom 群 · `给 jackie 建待办，约一下 global 线上…` → 归属 GlobCom，无建档 fields

**现象 / 常问**  
`jackie的待办有哪些` → Bot 说暂无；库里其实有 8 条。

**根因**  
`list_todos` 无 `assigneeName` 参数，模型把人名当 `partnerName` 查。

**改动点**  
- `src/lib/skills.ts` — `assigneeName` 筛选  
- `src/lib/assistant-core.ts` — 提示：某人待办用 assigneeName  

**回归注意**  
- `jackie的待办` 必须 **query.list_todos**，不是 `intake.todo`  
- 创建待办「给 jackie 建个待办」仍走 **intake.todo**
